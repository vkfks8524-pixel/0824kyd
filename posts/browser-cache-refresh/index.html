브라우저 문제 해결

웹페이지가 수정됐는데 예전 화면이 계속 보일 때 해결 방법

작성일: 2026년 7월 12일 · 예상 읽기 시간: 8분

홈페이지 파일을 수정했는데도 브라우저에는 예전 화면이 계속 보일 때가 있습니다. 이때는 실제 수정이 안 된 것인지, 브라우저 캐시가 예전 파일을 보여주는 것인지 먼저 구분해야 합니다.

1. 새로고침부터 다시 하기

가장 먼저 일반 새로고침을 합니다. Windows에서는 보통 F5 또는 Ctrl + R을 누릅니다. 그래도 바뀌지 않으면 강력 새로고침을 시도합니다.

Windows Chrome / Edge: Ctrl + F5
Windows Chrome / Edge: Ctrl + Shift + R

강력 새로고침은 브라우저가 저장해둔 일부 파일을 다시 받아오게 만드는 방법입니다.

2. 시크릿 창에서 확인하기

시크릿 창은 기존 로그인 정보와 캐시 영향을 덜 받습니다. Chrome이나 Edge에서 시크릿 창을 열고 같은 주소를 입력해봅니다. 시크릿 창에서는 새 화면이 보이는데 일반 창에서는 예전 화면이 보인다면 캐시 문제일 가능성이 큽니다.

3. 주소가 정확한지 확인하기

비슷한 주소를 여러 개 쓰는 경우가 있습니다. 예를 들어 kyd.kr과 www.kyd.kr, pages.dev 주소가 각각 다르게 보일 수 있습니다. 어느 주소를 대표 주소로 쓸지 정하고, 그 주소에서 확인해야 합니다.

4. 배포가 끝났는지 확인하기

Cloudflare Pages나 GitHub Pages처럼 자동 배포를 쓰는 경우, 파일을 저장했다고 바로 사이트에 반영되는 것은 아닙니다. 배포 작업이 끝나야 실제 페이지가 바뀝니다. Pages 대시보드에서 최근 배포가 성공 상태인지 확인합니다.

5. CSS만 안 바뀌는 경우

글 내용은 바뀌었는데 디자인만 그대로라면 CSS 파일 캐시가 남아 있을 수 있습니다. 이때도 강력 새로고침을 먼저 시도합니다. 개발 중이라면 CSS 파일 이름을 바꾸거나, 배포 도구가 파일 버전을 자동으로 붙이는 구조를 사용할 수도 있습니다.

정리

예전 화면이 계속 보일 때는 “수정 실패”라고 단정하지 말고 새로고침, 시크릿 창, 주소 확인, 배포 상태 확인 순서로 점검하는 것이 좋습니다. 이 순서만 지켜도 대부분의 혼란을 줄일 수 있습니다.

캐시 문제와 배포 문제를 구분하는 법

웹페이지가 바뀌지 않을 때는 브라우저 캐시만 의심하기 쉽지만, 실제로는 배포가 아직 끝나지 않았거나 다른 주소를 보고 있는 경우도 많습니다. 같은 사이트라도 루트 도메인, www 주소, pages.dev 주소가 각각 다르게 보일 수 있습니다.

가장 좋은 확인 방법은 세 가지 화면을 나눠 보는 것입니다. 일반 브라우저 창, 시크릿 창, 스마트폰 모바일 데이터 환경에서 같은 주소를 열어봅니다. 한 곳에서만 예전 화면이 보이면 캐시 가능성이 크고, 모든 환경에서 예전 화면이면 배포나 파일 경로 문제일 가능성이 큽니다.

초보자가 자주 놓치는 부분

HTML 파일은 바뀌었는데 CSS만 그대로라면 디자인이 바뀌지 않은 것처럼 보일 수 있습니다. 이때는 페이지 내용의 문장 하나를 일부러 바꿔서 실제 HTML이 반영됐는지 확인합니다. 문장은 바뀌었는데 색상이나 간격만 그대로라면 CSS 캐시를 의심합니다.

Cloudflare Pages처럼 자동 배포를 쓰는 경우에는 GitHub에 파일을 올린 것과 실제 사이트 반영이 같은 순간에 일어나지 않습니다. 배포 기록에서 가장 최근 커밋이 성공으로 표시되는지 확인해야 합니다.

확인 체크리스트

주소창의 URL이 대표 주소인지, 강력 새로고침을 했는지, 시크릿 창에서도 같은지, 모바일에서도 같은지, 배포 기록이 성공인지 순서대로 확인합니다. 이 다섯 가지를 점검하면 대부분의 혼란을 줄일 수 있습니다.

문제가 해결된 뒤에는 대표 주소를 하나로 정해두는 것이 좋습니다. 예를 들어 www 주소를 대표로 쓰기로 했다면 다른 주소는 모두 www 주소로 이동하게 설정해야 검색엔진과 방문자 모두 같은 페이지를 보게 됩니다.

실제 확인 예시

홈페이지 제목을 바꿨는데 계속 이전 제목이 보인다면, 먼저 페이지 본문에 임시로 눈에 띄는 문장을 하나 넣어봅니다. 그 문장이 보이면 HTML은 배포되었고 제목이나 스타일만 캐시되어 있을 가능성이 있습니다. 문장도 보이지 않으면 배포 자체가 반영되지 않았을 수 있습니다.

반대로 스마트폰에서는 새 화면이 보이는데 노트북에서만 예전 화면이 보인다면 노트북 브라우저 캐시 문제일 가능성이 큽니다. 이 경우 브라우저 데이터를 모두 삭제하기 전에 해당 사이트의 캐시만 지우거나 강력 새로고침을 먼저 시도하는 편이 덜 번거롭습니다.

기록해두면 좋은 정보

사이트를 운영한다면 수정한 시간, 커밋 번호, 배포 완료 시간, 확인한 주소를 간단히 기록해두면 좋습니다. 나중에 같은 문제가 생겼을 때 실제로 파일이 바뀌지 않은 것인지, 브라우저가 예전 파일을 보여준 것인지 더 빨리 판단할 수 있습니다.

특히 애드센스 코드, ads.txt, sitemap.xml처럼 외부 서비스가 확인하는 파일은 대표 주소에서 직접 열어보는 습관이 필요합니다. 브라우저 주소창에 파일 경로를 직접 입력해 보이는지 확인하면 설정 문제를 빠르게 발견할 수 있습니다.
